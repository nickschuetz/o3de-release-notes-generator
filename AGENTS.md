# AGENTS.md

## Project overview

O3DE Release Notes Generator ‚Äî a standalone Python tool that generates Open 3D Engine release notes from merged pull requests. Zero external dependencies (Python 3.10+ stdlib only).

## Build & run

```bash
# No build step ‚Äî single-file Python script
python release_notes.py --version

# Run tests
python -m pytest tests/ -v

# Regenerate SBOM
python generate_sbom.py
```

## Key commands

```bash
# Generate release notes (most common usage)
python release_notes.py generate \
  --from-ref <start-tag> --to-ref <end-branch> \
  --default-repo-path /path/to/o3de \
  --output-json release_data.json \
  --output-md release_notes.md \
  --release-version <version>

# Fetch only (JSON output for programmatic use)
python release_notes.py fetch \
  --from-ref <start-tag> --to-ref <end-branch> \
  --default-repo-path /path/to/o3de \
  --output-json release_data.json

# Render only (from existing JSON)
python release_notes.py render \
  --input-json release_data.json \
  --output-md release_notes.md \
  --release-version <version>
```

## Code conventions

- Zero external dependencies ‚Äî stdlib only, do not add pip packages
- All subprocess calls use list arguments, never `shell=True`
- All subprocess calls pass `encoding='utf-8', errors='replace'`
- All user inputs validated with regex before use in subprocess or file I/O
- GraphQL queries use server-side variables (`$owner`, `$name`); never string-interpolate owner/repo into the query body
- Atomic file writes via `tempfile.mkstemp()` + `os.replace()`
- Logging via `logging.getLogger('o3de.release_notes')` ‚Äî never log secrets; subprocess stderr passes through `_safe_stderr()` (token-scrub + truncate)
- Return codes: 0 = success, 1 = failure
- PR titles sanitized for markdown injection before rendering; PR bodies capped at 64KB before extraction
- Tests use `pytest` with `unittest.mock` ‚Äî no network calls in tests
- Summary command parsed via `shlex.split()`, not `.split()` ‚Äî supports quoted args
- Summary command runtime bounded by `--summary-timeout` (default 300s, range 10‚Äì3600s)
- LLM output cleaned by `_clean_summary()` ‚Äî strips preamble and dividers
- SIG categorization tiebreaks deterministically via `SIG_CANONICAL_ORDER` (label sort + title-keyword tiebreak) ‚Äî do not introduce non-deterministic ordering

## Architecture

Three-stage pipeline: Extract (git log) ‚Üí Categorize (SIG labels/heuristics) ‚Üí Render (markdown). See ARCHITECTURE.md for full details including security model (OWASP/NIST).

SIG categorization heuristics are data-driven dicts at the top of `release_notes.py` (`SIG_TITLE_KEYWORDS`, `SIG_FILE_PATH_PATTERNS`) ‚Äî edit these to adjust categorization.

## Things to know

- The o3de repo is always read-only ‚Äî this tool never writes to it
- `--repo-path` takes `owner/repo=/path` mappings for multi-repo support
- `--generate-summary` pipes a prompt via stdin to an LLM (default: `ollama run --nowordwrap qwen2.5:14b`; use `claude -p` for cloud, or `qwen2.5:32b` if you have ~24GB VRAM)
- `--summary-hint` injects narrative guidance into the LLM prompt ‚Äî inline text or `@filepath` to read from a file
- `--summary-timeout` bounds the LLM runtime (default 300s, range 10‚Äì3600s)
- `--dry-run` previews which PRs would be fetched (from local `git log`) without calling the GitHub API or writing files
- `--log-file PATH` appends logs to a file in addition to stderr
- JSON intermediate format supports `manual_override_sig` and `manual_override_description` fields ‚Äî these must be preserved on incremental re-runs
- **Watch for the merge drop warning**: PRs in the prior JSON that no longer appear in `git log` are dropped *unless* they carry a `manual_override_*` field. Direct edits to `description` / `sig_category` without setting the override are silently lost; `merge_with_existing` logs a `WARNING` count when this happens.
- Schema version is in `SCHEMA_VERSION` constant ‚Äî bump it when changing JSON structure, and accept the previous version for backward compatibility
- SBOM at `sbom.cdx.json` is auto-regenerated by GitHub Actions on push
- Tests run on Python 3.10 / 3.11 / 3.12 in CI on every push and PR (`.github/workflows/test.yml`)
- GitHub Actions are pinned to commit SHAs ‚Äî see CONTRIBUTING.md for the update policy
